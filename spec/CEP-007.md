# CEP-007: LLM Frontend Streaming Synchronization Engine

**Status:** Draft
**Layer:** User Interface / Streaming
**Authors:** Borja Moskv (borjamoskv / MOSKV-1 APEX)

---

## 1. Abstract

La propuesta **CEP-007** define la arquitectura e implementación del motor de sincronización de estado de renderizado para streams de texto de Modelos de Lenguaje Grande (LLMs) en interfaces frontend. 

El principal desafío en interfaces de chat con streaming continuo es el acoplamiento degenerativo entre el flujo de entrada de alta frecuencia y los ciclos de cálculo del motor visual del navegador. Esta especificación propone desacoplar la ingesta de I/O, el procesamiento incremental de AST (Árbol de Sintaxis Abstracta) y las actualizaciones del DOM, resolviendo de manera determinista los problemas de no-localidad en el parsing de Markdown, la estabilización de scroll frente a alturas dinámicas, la concurrencia de eventos de entrada y el procesamiento pesado de KaTeX y Shiki (resaltado de sintaxis).

## 2. Concurrency Matrix (Los 3 Relojes)

El motor de sincronización descarta el flujo lineal síncrono. El estado de la interfaz se gobierna por tres relojes independientes con niveles rígidos de prioridad, evitando que el renderizado de texto bloquee la interacción directa del usuario.

```text
                  ┌───────────────────────────────┐
                  │ Reloj 0: Entrada de Usuario   │◄─── Prioridad Máxima (Baja Latencia)
                  └───────────────┬───────────────┘
                                  ▼
                  ┌───────────────────────────────┐
                  │ Reloj 1: Mutación del DOM     │◄─── Prioridad Media (Max 30-60 Hz)
                  └───────────────┬───────────────┘
                                  ▼
                  ┌───────────────────────────────┐
                  │ Reloj 2: Ingesta I/O (Stream) │◄─── Prioridad Baja (100-1000 t/s)
                  └───────────────────────────────┘
```

### 2.1 Especificación de la Matriz

| Reloj | Entidad | Frecuencia típica | Primitivas de Control | Comportamiento |
| :--- | :--- | :--- | :--- | :--- |
| **Reloj 0** | Interacción Humana | Asíncrona (Event-driven) | `navigator.scheduling.isInputPending()`, `scheduler.yield()` | Interrumpe cualquier ciclo de parsing o commit si hay eventos de puntero, teclado o scroll pendientes. |
| **Reloj 1** | Mutación Visual | Ajustada a V-Sync (30-60 Hz) | `requestAnimationFrame`, Coalescencia Adaptativa | Agrupa modificaciones visuales en commits discretos por frame visual. Previene el layout thrashing. |
| **Reloj 2** | Ingesta I/O | 100-1000 tokens/segundo | Piece Table Append-only, Buffering temporal | Almacena los fragmentos crudos en memoria sin procesar el DOM de forma síncrona en cada carácter. |

## 3. Estructura de Memoria y Estado

El motor mantiene una separación estricta entre el flujo de bytes crudos y la representación espacial del documento en pantalla.

### 3.1 Estructura del Estado Lógico (TypeScript)

Para evitar la penalización por reasignación de cadenas en JavaScript ante streams masivos, se implementa una estructura tipo Piece Table combinada con un set de invalidación incremental.

```typescript
type BlockId = string;

enum BlockType {
  Paragraph = "Paragraph",
  Code = "Code",
  Math = "Math",
  List = "List"
}

interface EpistemicBlock {
  id: BlockId;
  type: BlockType;
  rawStartIndex: number;
  rawEndIndex: number;
  hash: string;
}

interface PieceTable {
  original: string;
  addBuffer: string;
  entries: Array<{
    source: "original" | "add";
    start: number;
    length: number;
  }>;
}

interface StateEngine {
  buffer: PieceTable;
  blocks: Map<BlockId, EpistemicBlock>;
  dirtySet: Set<BlockId>;
}
```

### 3.2 Estabilidad O(log N) de Alturas (Fenwick Tree)

Cuando se renderizan bloques con alturas desconocidas, el cálculo del offset de scroll requiere sumas de prefijos rápidas. Se implementa un **Fenwick Tree** (Binary Indexed Tree) para resolver consultas de offset y actualizaciones de altura en tiempo \(O(\log N)\).

```typescript
class HeightMap {
  private tree: Float64Array;
  private size: number;

  constructor(size: number) {
    this.size = size;
    this.tree = new Float64Array(size + 1);
  }

  public updateHeight(index: number, deltaHeight: number): void {
    let idx = index + 1; // Fenwick es 1-indexed
    while (idx <= this.size) {
      this.tree[idx] += deltaHeight;
      idx += idx & -idx;
    }
  }

  public getOffset(index: number): number {
    let sum = 0;
    let idx = index; // Retorna offset de inicio (suma hasta index - 1)
    while (idx > 0) {
      sum += this.tree[idx];
      idx -= idx & -idx;
    }
    return sum;
  }
}
```

## 4. Subsystem Isolation & Off-Thread Parsing

Los componentes costosos de formateo visual se extraen fuera del hilo de ejecución principal (Main Thread) para evitar micro-stutters.

```text
[Main Thread] ──► Ingesta de Tokens ──► Incremental Parser
     ▲                                       │
     │ (Estructura de Retorno)               ▼ (KaTeX / Shiki)
     └─────────────────────────────── [Web Worker Pool]
```

### 4.1 Reglas de Aislamiento y Delegación

1.  **Matemáticas (KaTeX):** Todo bloque detectado con delimitadores de tipo display `$$` o expresiones en línea `$` de alta densidad (complejidad estimada superior a 5 macros no estándar) se delega a un Web Worker.
2.  **Resaltado de Código (Shiki/WASM):** El parsing sintáctico a través de Shiki o WASM se ejecuta exclusivamente en hilos paralelos. El Main Thread solo recibe la cadena HTML parseada y la inyecta mediante transiciones atómicas.
3.  **Markdown General:** Debido a problemas de no-localidad en el parsing de elementos anidados (listas e identaciones), el procesamiento estructural básico se ejecuta en el Main Thread pero de forma fragmentada en rebanadas de tiempo (Time-Slicing).

### 4.2 Esquema de Políticas de Caché

```typescript
interface CacheKey {
  hash: string;
  themeOrMacros: string;
}

class LRUMemoryCache<T> {
  private cache = new Map<string, T>();
  private maxEntries: number;

  constructor(maxEntries = 500) {
    this.maxEntries = maxEntries;
  }

  public get(key: CacheKey): T | undefined {
    const serializedKey = `${key.hash}:${key.themeOrMacros}`;
    const value = this.cache.get(serializedKey);
    if (value !== undefined) {
      // Refresh en LRU
      this.cache.delete(serializedKey);
      this.cache.set(serializedKey, value);
    }
    return value;
  }

  public set(key: CacheKey, value: T): void {
    const serializedKey = `${key.hash}:${key.themeOrMacros}`;
    if (this.cache.size >= this.maxEntries) {
      const oldestKey = this.cache.keys().next().value;
      if (oldestKey !== undefined) {
        this.cache.delete(oldestKey);
      }
    }
    this.cache.set(serializedKey, value);
  }
}
```

## 5. Control Visual y Mitigación de Churn

### 5.1 Virtualización CSS via Content Visibility
Para evitar el impacto en el rendimiento debido al crecimiento del DOM (DOM Churn), los bloques no visibles del chat se contienen utilizando propiedades CSS modernas que aíslan el árbol visual sin dañar el modelo de objetos de accesibilidad (AOM):

```css
.semantic-block {
  content-visibility: auto;
  contain: layout paint style;
  contain-intrinsic-size: auto var(--estimated-block-height, 120px);
}

.scroll-container {
  overflow-anchor: auto;
}
```

### 5.2 Algoritmo de Compensación Dinámica de Viewport
Si la altura real de un bloque mutado dinámicamente (por ejemplo, al renderizar código o matemáticas diferidas) difiere del tamaño estimado, se ejecuta una compensación aritmética síncrona en el scroll:

```typescript
class ViewportStabilizer {
  private heightMap: HeightMap;
  private container: HTMLElement;

  constructor(container: HTMLElement, size: number) {
    this.container = container;
    this.heightMap = new HeightMap(size);
  }

  public registerBlockResize(blockIndex: number, realHeight: number, estimatedHeight: number): void {
    if (realHeight === estimatedHeight) {
      return;
    }

    const delta = realHeight - estimatedHeight;
    const blockTopOffset = this.heightMap.getOffset(blockIndex);
    const currentScrollTop = this.container.scrollTop;

    // Si el bloque está por encima del viewport actual, compensamos
    if (blockTopOffset < currentScrollTop) {
      this.container.scrollTop += delta;
    }

    // Actualizamos el mapa de alturas
    this.heightMap.updateHeight(blockIndex, delta);
  }
}
```

### 5.3 Lógica Stick-to-Bottom Activa
Se utiliza un `IntersectionObserver` sobre un nodo centinela al final del contenedor. El auto-scroll se detiene automáticamente ante interacciones humanas específicas para no interrumpir al usuario.

```typescript
class ScrollSentinel {
  private sentinel: HTMLElement;
  private container: HTMLElement;
  private isUserAtBottom = true;
  private observer: IntersectionObserver;

  constructor(container: HTMLElement, sentinel: HTMLElement) {
    this.container = container;
    this.sentinel = sentinel;

    this.observer = new IntersectionObserver((entries) => {
      for (const entry of entries) {
        this.isUserAtBottom = entry.isIntersecting;
      }
    }, {
      root: this.container,
      threshold: 1.0
    });

    this.observer.observe(this.sentinel);
  }

  public evaluateScrollTrigger(): void {
    const selection = window.getSelection();
    
    // Si el usuario está realizando una selección de texto, abortamos el autoscroll
    if (selection && selection.type === "Range") {
      return;
    }

    // Si la bandera indica que el usuario no ha subido manualmente, forzamos scroll
    if (this.isUserAtBottom) {
      this.container.scrollTop = this.container.scrollHeight;
    }
  }
}
```

## 6. Commit Scheduler (El Ciclo Principal)

El ciclo de ejecución principal coordina los tres relojes utilizando fragmentación de procesamiento (Time-Slicing) y cesión de control voluntaria.

```typescript
class CommitScheduler {
  private pendingUpdates: Array<string> = [];
  private isProcessing = false;

  public appendStreamChunk(chunk: string): void {
    this.pendingUpdates.push(chunk);
    this.triggerProcessingLoop();
  }

  private triggerProcessingLoop(): void {
    if (this.isProcessing) return;
    this.isProcessing = true;
    requestAnimationFrame(async () => {
      await this.processQueue();
    });
  }

  private async processQueue(): Promise<void> {
    const deadline = performance.now() + 16.6; // Límite estricto de 1 frame visual (16.6ms)

    while (this.pendingUpdates.length > 0) {
      // Ceder el hilo al navegador si hay entradas críticas de usuario pendientes
      if (navigator.scheduling?.isInputPending?.()) {
        await scheduler.yield();
      }

      const chunk = this.pendingUpdates.shift();
      if (chunk) {
        this.applyStateChange(chunk);
      }

      // Detener procesamiento en este frame si excedemos el presupuesto de tiempo
      if (performance.now() > deadline) {
        break;
      }
    }

    this.isProcessing = false;
    
    // Si quedan elementos en la cola, planificamos para el siguiente frame

      this.triggerProcessingLoop();
```

## 7. Matriz de Trade-offs y Decisiones de Diseño

| Alternativa Evaluada | Ventajas | Desventajas | Decisión |
| :--- | :--- | :--- | :--- |
| **Fenwick Tree vs Array Simple** | Suma de prefijos en \(O(\log N)\) frente a \(O(N)\). | Mayor complejidad matemática y consumo mínimo de memoria inicial. | **Fenwick Tree:** Necesario para chats extremadamente largos con miles de bloques. |
| **Off-Thread vs Main Thread Parsing** | Evita latencias visuales prolongadas al procesar texto. | Costo de serialización/deserialización a través del canal del Worker. | **Híbrido:** Markdown estándar en Main Thread, KaTeX/Shiki pesado en Workers. |
| **Virtualización DOM vs CSS containment** | Libera totalmente la memoria del DOM eliminando nodos. | Pierde soporte del AOM nativo y capacidad de búsqueda en la página. | **CSS Containment:** Se opta por `content-visibility: auto` para mantener el árbol accesible indexado. |
