# BABYLON-60 Proof IR Specification

## 1. Introduction
The Proof Intermediate Representation (Proof IR) is a language-agnostic formal structure. It explicitly models the causal trace of the execution, decoupling the runtime engine from the formal theorem provers (Lean 4, Coq). 

## 2. Structural Definition

A program in Proof IR is modeled as a tuple:
`P = (Σ, Λ, Π)`

Where:
- `Σ` is the sequence of state transitions `(σ_i, σ_{i+1})`.
- `Λ` is the logical causal ledger of emitted events.
- `Π` is the set of explicit proof obligations to verify structural invariants (e.g., `∀ e ∈ Λ, producer(e) = deterministic`).

## 3. Abstract Syntax Tree (AST) Mapping

The Proof IR is generated by parsing the `SSA + Typed IR` output of the `.b60` compiler. It retains strictly the mathematical effects of the instructions.

### 3.1. States (`σ`)
A state `σ` represents the exact values of the Abstract Machine tuple `M = (R, H, L, PC, Q, C)`. The transition relation `σ → σ'` represents the effect of a single deterministic operation.

### 3.2. Verification Goals
Each transition generates an implicit goal that must be proven in the backend.
For instance, for every division `BA.EXACT`, the Proof IR inserts a verification goal `divisor ≠ 0` into `Π`.

## 4. Backend Translation
The Proof IR is exported as a serialized S-Expression or JSON file (e.g., `proof.ir`), which a simple parser in Lean or Coq can read to construct the native definitions and lemmas automatically.
