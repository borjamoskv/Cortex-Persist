# CEP-001: Core Epistemic Microkernel Architecture

**Status:** Frozen Normative Draft
**Layer:** Core Kernel (C5-REAL)
**Authors:** CORTEX Autonomous Design Swarm (borjamoskv / MOSKV-1 APEX)

---

## 1. Abstract

La propuesta **CEP-001** define la arquitectura fundamental del microkernel de CORTEX. Establece que el núcleo del sistema **no ejecuta razonamiento, inferencia ni tasación de la verdad**. Su única responsabilidad es actuar como garante criptográfico y estructural de los objetos epistemológicos, sus relaciones y la evolución histórica del conocimiento. El microkernel se diseña como un "Git epistémico" inmutable y verificable.

Todos los motores analíticos, LLMs, solvers lógicos y proveedores de confianza son relegados al rol de **Epistemic Drivers** (módulos externos conectables) que se comunican con el núcleo a través de un estricto Interfaz Binario de Aplicación (ABI).

---

## 2. Core Axioms & Kernel Responsibilities

El microkernel de CORTEX asume exclusivamente **cuatro responsabilidades primitivas e inmutables**:

1. **Immutable Objects (Data Layer):** Almacenamiento de entidades (Aserciones, Evidencias, Restricciones) indexadas por el hash criptográfico de su contenido (Content-Addressable Storage).
2. **Immutable Relations (Graph Layer):** Mantenimiento de la topología semántica mediante `SupportRelations` inmutables.
3. **Immutable States (Snapshot Layer):** Versionado del grafo global de conocimiento mediante un Merkle DAG histórico.
4. **Commit Validation (Boundary Layer):** El protocolo de certificación estructural (Gatekeeper) que previene la corrupción del DAG.

Cualquier operación que exceda estas responsabilidades (inferencia, resolución de conflictos, asignación de pesos de verdad, RAG) pertenece estrictamente al espacio de usuario (Drivers).

## 3. Core Object Model (Capa de Datos y Relaciones)

El conocimiento no se almacena como texto plano estocástico; se descompone en tipos atómicos inmutables.

### 3.1 EpistemicObject (Base)
Todo objeto que entra al kernel hereda de una primitiva inmutable, forzando la indexación por contenido.
* **ID:** `SHA3-256(canonical_serialization)`
* **Invariante:** Los objetos son *write-once, read-many*. No pueden modificarse, solo dereferenciarse en estados futuros.

### 3.2 Tipos Primitivos
* **`Assertion`**: Un reclamo lógico o fáctico sobre la realidad.
* **`Evidence`**: Un fragmento crudo de datos observables (documento, log, hash de imagen).
* **`Constraint`**: Una regla de invariancia lógica (ej. mutuamente excluyente).

### 3.3 SupportRelation (El Grafo Desacoplado)
Para preservar la inmutabilidad de los datos, la semántica de "cómo" se relacionan los objetos se extrae a una entidad separada.
* **Estructura:**
  * `evidence_id`: Puntero a la causa.
  * `assertion_id`: Puntero al efecto.
  * `relation_type`: Enum (`SUPPORTS`, `REFUTES`, `OBSERVES`, `DERIVED_FROM`, `CONTRADICTS`).

*Consecuencia:* Un cambio en la interpretación de una evidencia no altera el objeto Evidence; solo requiere la inserción de una nueva `SupportRelation` en el próximo estado.

## 4. Core State Model (EpistemicState)

El estado del sistema no es una base de datos mutable. Es un nodo inmutable dentro de un Grafo Acíclico Dirigido (DAG).

* **`state_id`**: UIDv7 (Ordenamiento temporal y unicidad).
* **`parent_states`**: Punteros a estados previos (Soporta bifurcaciones y uniones).
* **`merkle_root`**: Raíz criptográfica de los sub-árboles que componen el estado (Objetos, Relaciones, Valuaciones).

### El Invariante del TrustValue
Cada estado incluye un mapa opaco: `epistemic_valuations: Map<AssertionID, TrustValue>`. El kernel **no sabe calcular este valor**, solo garantiza su persistencia en el snapshot.

## 5. Epistemic Commit Protocol (El ABI del Kernel)

El sistema opera bajo un modelo de **Propuesta y Certificación**. El núcleo no muta activamente, actúa como un guardián de transacciones.

### 5.1 El Flujo de Transición
```text
[ Estocasticidad / IA ]            [ Invariante Estructural ]
Inference Engine Driver ─────────> Epistemic Commit (Kernel Gate)
(Genera Propuesta de Estado)       │
                                   │ (Auditoría Estructural Pura)
                                   ├─> ¿Hashes calculados son correctos?
                                   ├─> ¿Dangling references? (Huérfanos)
                                   ├─> ¿El DAG es acíclico?
                                   │
                                   ▼
                             Committed State (Inmutable)
```

### 5.2 Bifurcación del Commit
El sistema operativo admite que el conocimiento sea "almacenado" (válido) antes de ser "evaluado" (confiable).

1. **Logical Commit:**
   * El `ProposedState` pasa las validaciones topológicas y de tipado.
   * Se almacena y se ancla al DAG.
   * Su mapa de valuaciones de verdad está vacío o marcado como `UNKNOWN`.
2. **Trusted Commit:**
   * Un driver especializado (Trust Provider) lee un estado `UNKNOWN`.
   * Computa el álgebra de confianza en base al grafo de relaciones.
   * Emite un nuevo estado donde la única mutación es la inyección de `epistemic_valuations`.

---

## 6. Execution Interfaces (Arquitectura de Drivers)

Los motores cognitivos operan como "Drivers" desacoplados que interactúan con el Kernel a través del ABI de Commits.

* **Inference Engine Driver:** Lee un `CommittedState`, ejecuta lógica estocástica (LLM, parseo) y propone un `Logical Commit` con nuevos nodos y relaciones.
* **Trust Provider Driver:** Lee un `Logical Commit`, ejecuta algoritmos bayesianos o heurísticas y propone un `Trusted Commit` con el cálculo de la verdad.
* **Solver Driver:** Escanea el DAG buscando contradicciones (ej. SAT Solver) y propone un estado refutando aserciones inválidas.

---

## 7. Kernel Contracts & Panic Conditions

Cualquier violación del contrato estructural durante un `EpistemicCommit` no genera una advertencia; genera un **Kernel Panic**, abortando la transición atómica y revirtiendo la operación.

| Panic Code | Trigger | Consecuencia Termodinámica |
| :--- | :--- | :--- |
| `PANIC_ORPHAN_RELATION` | Referencia a `evidence_id` o `assertion_id` inexistente en el estado resultante. | Corrupción topológica abortada. |
| `PANIC_HASH_COLLISION` | El hash declarado difiere del hash derivado matemáticamente del contenido. | Inyección de entropía abortada. |
| `PANIC_CAUSAL_CYCLE` | Una cadena de relaciones `DERIVED_FROM` genera un bucle cerrado. | Paradoja epistémica bloqueada. |
| `PANIC_STATE_FORK_CONFLICT` | Transición paralela sin declaración explícita de `merge`. | Race condition mitigada. |
